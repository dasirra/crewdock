# Mission Control Integration Design

**Date:** 2026-03-12
**Status:** Draft
**Topic:** Integrating builderz-labs/mission-control with OpenClaw NAS

## Goal

Add a web-based dashboard (Mission Control) to OpenClaw NAS so the user can send tasks to agents and monitor sessions in real time from a browser, while keeping existing Telegram/Discord channels fully functional.

## Constraints

- Ease of independent updates is the top priority (MC and gateway update separately).
- Access is always via Tailscale (local and remote).
- MC must connect to the gateway via WebSocket for live session monitoring.
- Existing gateway setup (network_mode: host, Tailscale Serve, Telegram/Discord) must not change.

## Architecture

Two separate Docker containers on the same host, both using `network_mode: host`.

```
NAS Host (network_mode: host for both containers)

  ┌──────────────────────┐     ┌──────────────────────┐
  │  openclaw-gateway    │     │  mission-control      │
  │  Port: 18789         │◄────│  Port: 3000           │
  │  Bridge: 18790       │ WS  │  WS → localhost:18789 │
  │                      │     │                       │
  │  LLM Agent (24/7)    │     │  Next.js Dashboard    │
  │  Telegram Bot        │     │  SQLite (tasks)       │
  │  Discord Bot         │     │  REST API + SSE       │
  │  Forge (cron)        │     │                       │
  │  Claude Sessions     │     │  Vol: mc-data (DB)    │
  │                      │     │  Vol: config/openclaw  │
  │  Vol: config/openclaw│     │       (read-only)     │
  │  Vol: workspace      │     └───────────────────────┘
  │  Vol: projects       │
  │  Vol: config/claude  │
  └──────────────────────┘

  ┌─────────────────────────────────────────────────────┐
  │  Tailscale Serve (HTTPS :443)                       │
  │  /         → 127.0.0.1:3000  (Mission Control)     │
  │  /gateway  → 127.0.0.1:18789 (OpenClaw Gateway)    │
  └─────────────────────────────────────────────────────┘
```

### Why separate containers

- Independent update cycles: updating MC does not touch the gateway image, and vice versa.
- Fault isolation: if MC crashes, the gateway and all agents keep running.
- Clean separation of concerns: MC is a Next.js app with its own SQLite DB; gateway is the OpenClaw runtime.

### Why not same container

- Updates become coupled: any change requires rebuilding and restarting both services.
- The gateway base image (`alpine/openclaw`) is third-party controlled. Adding MC inside it creates fragile layering.
- Process supervision (s6, supervisord) adds operational complexity.

## Docker Configuration

### Source management

Mission Control source lives as a git submodule:

```
openclaw-nas/
  vendor/
    mission-control/    # git submodule → github.com/builderz-labs/mission-control
```

There is no published container image for mission-control, so it must be built from source.

### docker-compose.yaml (new service)

```yaml
mission-control:
  container_name: mission-control
  build:
    context: ./vendor/mission-control
    dockerfile: Dockerfile
  restart: unless-stopped
  network_mode: host
  environment:
    - PORT=3000
    - AUTH_USER=${MC_AUTH_USER:-admin}
    - AUTH_PASS=${MC_AUTH_PASS}
    - API_KEY=${MC_API_KEY}
    - OPENCLAW_HOME=/home/nextjs/.openclaw
    - OPENCLAW_GATEWAY_HOST=127.0.0.1
    - OPENCLAW_GATEWAY_PORT=18789
    - OPENCLAW_GATEWAY_TOKEN=${OPENCLAW_GATEWAY_TOKEN}
    - NEXT_PUBLIC_GATEWAY_URL=wss://${TS_CERT_DOMAIN}/gateway
    - MC_ALLOWED_HOSTS=localhost,127.0.0.1,${TS_CERT_DOMAIN}
  volumes:
    - mc-data:/app/.data
    - ./config/openclaw:/home/nextjs/.openclaw:ro
    - ./workspace:/home/nextjs/.openclaw/workspace:ro
```

### Volumes

| Volume | Purpose | Mode |
|--------|---------|------|
| `mc-data` | MC's SQLite database (tasks, agents, config) | read-write |
| `./config/openclaw` | Gateway config, for MC's Memory Browser and log viewer | read-only |
| `./workspace` | Agent workspace, for MC's agent memory browsing | read-only |

### Environment variables (new in .env.example)

```bash
# Mission Control
MC_AUTH_USER=admin
MC_AUTH_PASS=               # auto-generated by setup.sh (32-char random)
MC_AUTH_PASS=               # auto-generated by setup.sh (32-char random)
MC_API_KEY=                 # auto-generated by setup.sh (64-char random)
TS_CERT_DOMAIN=             # your Tailscale node FQDN (e.g., nas.tailnet-name.ts.net)
```

`OPENCLAW_GATEWAY_TOKEN` is already in `.env` and shared with the MC service for gateway authentication.

## Tailscale Serve Configuration

Single HTTPS domain with path-based routing.

### tailscale-serve.json

```json
{
  "TCP": {
    "443": { "HTTPS": true }
  },
  "Web": {
    "${TS_CERT_DOMAIN}:443": {
      "Handlers": {
        "/gateway": {
          "Proxy": "http://127.0.0.1:18789"
        },
        "/": {
          "Proxy": "http://127.0.0.1:3000"
        }
      }
    }
  }
}
```

Order matters: `/gateway` (more specific) before `/` (catch-all).

### WebSocket routing

MC's browser client connects to the gateway WebSocket via `wss://${TS_CERT_DOMAIN}/gateway`. This works for both local and remote access since all traffic goes through Tailscale.

`NEXT_PUBLIC_GATEWAY_URL=wss://${TS_CERT_DOMAIN}/gateway` is set at build/runtime so the browser always uses the Tailscale endpoint.

## Gateway Configuration

Changes to `config/openclaw/openclaw.json`:

```json
{
  "gateway": {
    "controlUi": {
      "allowedOrigins": [
        "https://${TS_CERT_DOMAIN}",
        "http://localhost:3000"
      ]
    },
    "auth": {
      "allowTailscale": true
    }
  }
}
```

- `allowedOrigins`: CORS policy. Without this, browsers block the WebSocket connection from MC to the gateway. The Tailscale domain is the primary origin; `localhost:3000` is a fallback for local debugging.
- `auth.allowTailscale`: already configured, no change needed.

## Makefile Commands

### New commands

```makefile
setup-mc               # git submodule init + update, generate MC_AUTH_PASS and MC_API_KEY in .env
up-mc                  # build + start MC only
down-mc                # stop MC only
logs-mc                # tail MC logs
update-mc              # git pull submodule + rebuild + restart MC
shell-mc               # bash into MC container
```

### Modified commands

```makefile
setup                  # now also runs setup-mc
up                     # starts gateway + MC
down                   # stops all services
update                 # updates gateway (check Docker Hub) then MC (pull submodule + rebuild), independently
```

`make update` runs both updates sequentially. If one fails, the other service keeps running.

### Setup flow (first run)

`make setup` will:
1. Run existing gateway setup (create .env, dirs, install agents, init forge.db)
2. Run `git submodule update --init vendor/mission-control`
3. Generate `MC_AUTH_PASS` (32-char random) and `MC_API_KEY` (64-char random), append to `.env`
4. Prompt user to set `TS_CERT_DOMAIN` in `.env`

## Communication Flows

### Browser (Mission Control)

```
Browser → https://nas.tailnet/ → MC (:3000) → WS to Gateway (:18789)
```

Task creation, live session monitoring, agent management, memory browsing.

### Telegram / Discord (unchanged)

```
Message → Gateway → Forge → Claude sessions
```

Commands: run, status, history, etc. No changes to existing flows.

### Forge cron (unchanged)

```
Cron (15 min) → Forge → GitHub issues → spawn autopilot sessions
```

MC can observe these sessions in real time via the gateway WebSocket.

## Update Procedures

### Update OpenClaw Gateway

```bash
make update
# or just the gateway:
# Checks Docker Hub for new alpine/openclaw version, rebuilds if newer
```

No impact on MC.

### Update Mission Control

```bash
make update-mc
# Equivalent to:
# cd vendor/mission-control && git pull origin main && cd ../..
# docker compose build mission-control
# docker compose up -d mission-control
```

No impact on gateway.

### Pin MC version

To pin MC to a specific version instead of tracking main:

```bash
cd vendor/mission-control
git checkout v1.2.3    # or a specific commit
cd ../..
git add vendor/mission-control
git commit -m "chore: pin mission-control to v1.2.3"
```

The submodule pointer in git tracks the exact commit.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Tailscale Serve path routing doesn't support WebSocket upgrade on `/gateway` | Test early. Fallback: use port-based routing (`:443` for MC, `:8443` for gateway) |
| MC Dockerfile breaks on NAS architecture (ARM vs x86) | Verify NAS CPU arch matches MC's Node.js build target. `better-sqlite3` needs native compilation. |
| `NEXT_PUBLIC_GATEWAY_URL` is baked at build time (Next.js) | Rebuild MC when changing Tailscale domain. Or check if MC supports runtime env override. |
| MC submodule drifts from upstream (breaking changes) | Pin to stable tags/commits instead of tracking main. Review changelogs before updating. |
